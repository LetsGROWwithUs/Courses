Overview & Architecture

SAP Datasphere is SAP's cloud-native, unified data platform that combines data integration, warehousing, federation, and semantic modeling — all on SAP Business Technology Platform (BTP).

What is SAP Datasphere?

Launched in 2023 as the evolution of SAP Data Warehouse Cloud (DWC), SAP Datasphere is not just a data warehouse — it is a Business Data Fabric. The core philosophy is to connect and harmonize data from any source (SAP or non-SAP, cloud or on-premise) without necessarily copying it, and to expose it in a semantically meaningful, governed, and business-ready way.

It runs entirely on SAP HANA Cloud as the in-memory database engine, hosted on SAP BTP. This means you get the full power of HANA (columnar storage, in-memory processing, calculation engine) without managing infrastructure.

Business Data Fabric: SAP's vision where data from all enterprise sources — S/4HANA, BW, third-party systems — is accessible through a unified semantic layer, regardless of where it physically resides. Datasphere is the technical implementation of this vision.

Deployment Model

Datasphere is a fully managed cloud service — SAP handles infrastructure, patching, scaling, and upgrades. You provision it through SAP BTP Cockpit as a subscription. There is no on-premise version of Datasphere itself, though it can connect to on-premise sources via the Data Provisioning Agent.

Each customer gets a Tenant — a dedicated Datasphere instance. Within the tenant, workloads are organized into Spaces.

Spaces & Tenants

Spaces are the fundamental isolation unit in Datasphere — every object, user, connection, and data lives within a Space.

What is a Space?

A Space is an isolated, self-contained workspace within a Datasphere tenant. Think of it as a dedicated project room — it has its own storage allocation, compute resources, users, connections, and schema. Different business units, projects, or environments (Dev/QA/Prod) typically get separate Spaces.

Analogy: If Datasphere is a building, a Space is a secure room with its own key. Occupants of Room A cannot enter Room B unless explicitly given access — but the building admin can grant cross-room passes (cross-space sharing).

Objects in one space can be shared to other spaces. The sharing is always read-only — the consuming space cannot modify the shared object. This enables a hub-and-spoke model where a central data space houses master data that other project spaces consume.

Every Space is backed by a HANA Cloud schema with the same name as the Space ID. You can expose this via Open SQL Schema to allow external JDBC/ODBC connections from tools like DBeaver, SAP HANA Studio, or Tableau.

Tables

Tables are the foundation of data storage in Datasphere. Understanding the three table types and their behaviors is essential for any modeling task.

Local Tables

A Local Table is a table physically stored inside Datasphere's HANA Cloud schema. Data is written into this table and persisted in Datasphere. It is the only table type you can write data into.

  • Created by: Defining structure manually or importing CSV/data flow output
  • Storage: Consumes your Space's disk quota
  • Editable: Yes — you can add/delete rows, update data
  • Used for: Planning data write-back, intermediate staging, master data uploads, target of Data Flows

Local Table — Record Flags

Local tables support technical delta tracking. Each row can have a record flag indicating its change type:

FlagMeaning
NNew — inserted record
UUpdated — modified record
DDeleted — soft-deleted record (still physically present)
XReverse image — used in delta processing

Remote Tables

A Remote Table is a virtual representation of a table from a connected source system. The data physically stays at the source — Datasphere creates a metadata pointer to it. When you query a remote table, the query is pushed down to the source system.

  • Created by: Importing table metadata from a connected source
  • Storage: Zero — data stays at source
  • Editable: No — read-only
  • Performance: Depends on source system and network latency
  • Used for: Federation (accessing live source data), initial exploration before deciding to replicate

Replica Tables

A Replica Table is a remote table that has been replicated — its data has been copied into Datasphere's local HANA storage. It is still "owned" by the remote source but has a local copy for performance.

The progression: Remote Table (virtual, slow) → enable replication → Replica Table (copy in HANA, fast). You can switch between access modes in the remote table settings.

Partitioning

Local tables support table partitioning to optimize query performance for large datasets. Partitioning splits the table into smaller physical segments based on a partition key.

  • Hash partitioning — distributes rows evenly across partitions
  • Range partitioning — by date/value range (e.g. one partition per year)
  • Queries that filter on the partition key only scan relevant partitions — significantly faster for large tables

Views are virtual objects that transform and combine data without storing it. They are the backbone of semantic modeling in Datasphere.

View Types

Visual drag-and-drop canvas with transformation nodes. No SQL coding needed. Ideal for modelers who prefer visual tools. Internally generates SQL.

SQL View — Key Syntax Features

Datasphere SQL Views support SAP HANA SQL dialect (Standard SQL mode). Key supported features:

-- Basic join
SELECT "Products"."PRODUCTID", "ProductTexts"."SHORT_DESCR"
FROM "Products"
INNER JOIN "ProductTexts" 
  ON "Products"."PRODUCTID" = "ProductTexts"."PRODUCTID"
  AND "ProductTexts"."LANGUAGE" = 'EN';

-- Window function
SELECT SALESORDERID, AMOUNT,
  SUM(AMOUNT) OVER (PARTITION BY CUSTOMERID ORDER BY ORDERDATE) AS RUNNING_TOTAL
FROM "SalesOrders";

-- CASE expression
SELECT PRODUCTID,
  CASE WHEN PRICE > 1000 THEN 'Premium'
       WHEN PRICE > 100  THEN 'Mid-range'
       ELSE 'Budget' END AS PRICE_SEGMENT
FROM "Products";

Input Parameters

Views can define Input Parameters — runtime variables that filter data when the view is queried. For example, a LANGUAGE parameter that limits text records to one language.

  • Defined in the view properties panel
  • SAC prompts the user for input parameter values when opening a story based on such a view
  • Can have default values
  • Useful for multi-language models, fiscal year variants, company code filters

View Persistency

By default views are virtual — executed at query time. For complex views with many joins or aggregations, you can enable Persistency to materialize the view result as a snapshot in HANA storage.

ModeWhen computedPerformanceFreshness
Virtual (default)At every querySlower for complex viewsAlways current
PersistedOn schedule (or manually)Much fasterAs of last refresh

Expose for Consumption

The Expose for Consumption toggle (found in Model Properties) makes a view or analytical dataset available for SAC live connections and external JDBC/ODBC access. When OFF, the object is internal-only and invisible to SAC.

Common mistake: Building a complete view but forgetting to turn on "Expose for Consumption" — SAC will not show it as an available model. Also ensure the semantic type is Analytical Dataset (not Relational Dataset) for SAC direct connection.

Joins Deep Dive

Joins are how tables are combined in Datasphere. Understanding join types, their behavior, and when to use each is fundamental to correct data modeling.

INNER JOIN

Returns only rows that have a match in both tables. Non-matching rows from either side are dropped.

SELECT P."PRODUCTID", P."PRICE", T."SHORT_DESCR"
FROM "Products" P
INNER JOIN "ProductTexts" T ON P."PRODUCTID" = T."PRODUCTID";
Use when: You only want records that exist in both tables. If a product has no text, you don't want it in the result.

LEFT JOIN (LEFT OUTER JOIN)

Returns all rows from the left table plus matching rows from the right. Non-matching right rows → NULL.

SELECT P."PRODUCTID", P."PRICE", T."SHORT_DESCR"
FROM "Products" P
LEFT JOIN "ProductTexts" T ON P."PRODUCTID" = T."PRODUCTID";
Use when: You want all products regardless of whether they have a text entry. Missing text shows as NULL.

RIGHT JOIN

Returns all rows from the right table plus matching rows from the left. Non-matching left rows → NULL. Rarely used — usually rewritten as LEFT JOIN with tables swapped.

FULL OUTER JOIN

Returns all rows from both tables. NULLs fill in where there is no match on either side.

Performance note: FULL JOINs can be expensive on large tables. Use only when genuinely needed to preserve orphan records from both sides.

CROSS JOIN

Cartesian product — every row from left × every row from right. No ON condition. Rows = left_count × right_count.

-- 100 products × 12 months = 1,200 rows
SELECT P."PRODUCTID", M."MONTH"
FROM "Products" P CROSS JOIN "MonthDim" M;
Valid use case: Generating a planning grid (all product × period combinations) as a starting template. Otherwise almost never intentional.

Join Cardinality

In Datasphere's graphical view join node, you must specify cardinality — the expected relationship between the two tables. This helps the HANA optimizer generate better query plans.

CardinalityMeaningExample
1:1One row in each table matches at most one row in the otherEmployee ↔ EmployeeDetails
1:NOne row in left matches many rows in rightSalesOrder → SalesOrderItems
N:1Many rows in left match one row in rightSalesOrderItems → Product
N:MMany-to-many — avoid if possible, or resolve via a bridge tableProducts ↔ Suppliers

Multi-column Join Conditions

Joins can have multiple conditions using AND. A common example in Datasphere's sample data:

-- Ensures product text language matches category text language
INNER JOIN "ProductTexts" AS T 
  ON T."PRODUCTID" = P."PRODUCTID" 
  AND T."LANGUAGE" = C."LANGUAGE"

Without the LANGUAGE condition, each product row would join to all language variants of its text — multiplying rows unintentionally.

Join Ordering Impact

In nested joins (chained in graphical view), the order matters for two reasons:

  • A table can only reference columns from tables already joined above/before it in the chain
  • Join order affects intermediate result set sizes, which impacts performance
  • Moving a table to an earlier join point may lose reference to columns it needs (like LANGUAGE from a table joined later)
Practical example: ProductTexts joins on both PRODUCTID and LANGUAGE — the LANGUAGE comes from ProductCategoryTexts. If you move ProductTexts before ProductCategoryTexts in the join, the LANGUAGE reference breaks.

Analytic Model

The Analytic Model is the star-schema assembly layer — it combines an Analytical Dataset (fact) with Dimensions, Hierarchies, and Measures into a query-ready structure for SAC.

What is an Analytic Model?

An Analytic Model sits on top of an Analytical Dataset (your fact view) and assembles the complete analytical structure — dimensions, attributes, hierarchies, measures, filters, and currency settings. It is the primary object that SAC consumes for reports and stories.

BW equivalent: Analytic Model ≈ BEx Query / Analytical Query. It defines what the business user sees — which dimensions to slice by, which measures to aggregate, what hierarchies to drill through.

Data Flows & Pipelines

Flows are executable objects that move and transform data. Datasphere offers three flow types — each serving a distinct integration or transformation purpose.

Flow Type Comparison

Flow TypePurposeSource → TargetBW/ETL Equivalent
Replication FlowCopy data from external source into DatasphereExternal source → Local/Replica TableSLT / ODP Replication
Data FlowETL — extract, transform, load between any objectsAny source → Local TableSAP Data Services / BODS
Transformation FlowLightweight transformation between local objectsLocal Table → Local TableBW DTP + Transformation

Replication Flow

A Replication Flow connects to an external source and replicates tables — either as a one-time full load or continuously with delta (changes). It is the recommended way to bring source system data into Datasphere at scale.

  • Initial Load: Full copy of the source table
  • Delta: Continuous or scheduled replication of changes (INSERT/UPDATE/DELETE) using the source's change capture mechanism
  • Supported sources: S/4HANA, BW, ABAP systems, SAP HANA, ODP sources (BW extractors)
  • Multiple tables can be added to one Replication Flow — batch replication management
  • Target is always a Local Table (auto-created with matching structure)

Data Flow

A Data Flow is a visual ETL pipeline — similar to a mapping in SAP Data Services or an SSIS package. It has sources, transformation operators, and a target local table.

Data Flow Operators

OperatorFunction
JoinCombine two datasets during transformation
UnionStack rows from multiple sources
ProjectionSelect and rename columns
AggregationPre-aggregate data before loading
FilterFilter rows during transformation
ScriptCustom Python/SQL logic within the flow

Transformation Flow

A Transformation Flow is a lightweight, SQL-based flow that reads from one or more local tables or views and writes into a local table. It supports UPSERT (update if exists, insert if not) and Full Load modes.

When to use: Post-replication transformations — e.g. after replicating S/4HANA ACDOCA into Datasphere, run a Transformation Flow to derive a clean Financial Actuals table with only the required columns and calculations.

Scheduling

All flow types support scheduling via the built-in Datasphere scheduler:

  • One-time execution
  • Recurring: Hourly, Daily, Weekly, Monthly
  • Cron-based for precise scheduling
  • Flows can be chained — a completed replication triggers a transformation flow

Task Chains

A Task Chain is an orchestration object that sequences multiple flows, view persistency refreshes, or other tasks in a defined order with dependency handling — similar to a BW Process Chain.

FeatureDetails
Sequential executionTask B runs only after Task A succeeds
Parallel executionMultiple tasks run simultaneously when no dependency
Error handlingDefine behavior on failure — stop chain or continue
NotificationsEmail alerts on success or failure

Replication & Federation

Two fundamental data access patterns in Datasphere — understanding when to replicate vs federate is a key architectural decision.

Federation (Virtual Access)

In federation mode, data stays at the source. Datasphere creates a virtual (remote) table that, when queried, sends the query to the source system and returns results in real time. There is zero data copy and zero storage consumed in Datasphere.

  • Latency: Network + source system query time (can be slow)
  • Freshness: Always live — reflects the source at query time
  • Good for: Large infrequently queried tables, operational reporting where freshness > speed, initial exploration
  • Risk: Adds load to source system; source outage = no data in Datasphere

Replication (Physical Copy)

In replication mode, data is physically copied from the source into Datasphere's HANA Cloud storage. The replica table contains an up-to-date local copy of source data, refreshed on schedule or via delta capture.

  • Latency: Fast — query hits local HANA
  • Freshness: As of last replication (may be seconds to hours depending on mode)
  • Good for: Reporting, SAC stories, joined analytics, high-frequency queries
  • Cost: Consumes storage quota in the Space

Replication Modes

ModeHow it worksSource requirement
Initial Load (Full)One-time full copy of all recordsAny connected source
Real-Time ReplicationChanges captured immediately as they happen at sourceSource must support change data capture (SAP sources via ODP/SLT)
Scheduled ReplicationDelta loaded on a schedule (e.g. every hour)Source must provide delta capability
ODP (Operational Data Provisioning): SAP's framework for delta-enabled data extraction from ABAP systems. Replication Flows for SAP sources use ODP under the hood — it tracks which records changed since the last extraction and provides only those deltas.

Switching Between Modes

For a remote table, you can switch between Federation and Replication at any time in the remote table editor — under Data Access → Remote / Replicated. Switching to Replicated triggers an initial load to copy data into HANA.

Integration

Data Provisioning Agent (DP Agent)

The DP Agent is the secure bridge between Datasphere (cloud) and on-premise or private-network source systems.

Why DP Agent?

Datasphere runs in the cloud. On-premise systems (S/4HANA on-premise, BW, Oracle, SQL Server) are behind firewalls and not directly reachable from the internet. The DP Agent is a lightweight Java agent installed on a server in your on-premise network that:

  • Initiates an outbound HTTPS connection to Datasphere (no inbound firewall rules needed)
  • Acts as a proxy — Datasphere sends queries/commands to the agent, which executes them locally against the source
  • Returns results securely over the same HTTPS tunnel

DP Agent Architecture

Direction: DP Agent → Datasphere (outbound only). Your firewall only needs to allow outbound HTTPS from the agent server to Datasphere's URL. No inbound port opening required — a significant security advantage.

DP Agent Setup Steps

1
Download DP Agent installer

Download from SAP Software Center. Supported on Windows Server and Linux.

2
Install on on-premise server

Install on a server with network access to both the source system and the internet (for Datasphere).

3
Register agent in Datasphere

In Datasphere System → Configuration → Data Provisioning Agents, register the agent using its Agent Name and credentials.

4
Install adapters

DP Agent uses adapters for each source type (ABAP adapter for SAP BW/S4, JDBC adapter for Oracle/SQL Server). Install the required adapters on the agent.

5
Create connection in Datasphere

Create a connection in the Space that uses the DP Agent — specify the agent name in the connection settings.

Business Layer

Semantic Layer Overview

The Business Layer is Datasphere's semantic abstraction layer — it transforms technical data models into business-friendly structures that analysts can use without SQL knowledge.

Why a Semantic Layer?

Data Builder views are technically correct but often use technical column names like KBETR, WAERS, MENGE_ST. A business analyst working in SAC needs to see Rate, Currency, Quantity. The Business Layer bridges this gap — it adds business names, descriptions, relationships, and KPI logic on top of the technical model.

Two-layer architecture:
Layer 1 — Data Builder: Technical tables, views, SQL logic, joins
Layer 2 — Business Layer: Business names, measures, KPIs, associations, perspectives
SAC consumes Layer 2 (or directly Layer 1 Analytic Models).

Business Layer Objects Overview

ObjectThink of it asBW equivalent
Business EntityA semantically enriched view of a Data Builder objectInfoObject / DSO with business metadata
Consumption ModelQuery-ready model combining business entities for SACBEx Query / Analytical Query
PerspectiveA focused subset/view of a Consumption ModelBEx Query Variant / BW View

Associations

In the Business Layer, Associations define relationships between Business Entities — similar to foreign key relationships but declared at the semantic level rather than enforced at the database level.

For example: a SalesOrders business entity → associated to BusinessPartners via PARTNERID. When SAC queries through this model, HANA automatically resolves the join at query time.

Associations vs Data Builder Joins: Data Builder joins physically embed the SQL join in the view definition. Business Layer associations are metadata — the join is generated dynamically at query execution. This makes associations more flexible (different query tools can navigate differently) but slightly less predictable in behavior.
Business Layer

Business Entities

A Business Entity wraps a Data Builder object (table or view) with semantic annotations — business names, descriptions, key definitions, and KPIs.

Creating a Business Entity

You create a Business Entity on top of an existing Data Builder view or table. The entity inherits all columns from the source and you annotate them:

  • Business Name: User-friendly label (e.g. "Sales Amount" instead of "NETWR")
  • Description: Full definition visible in catalog and SAC
  • Key: Mark columns that uniquely identify a record
  • Attribute Type: Text, ID, Amount, Quantity, Date, etc.
  • Semantic Type: Fact, Dimension, Measure

KPIs (Key Performance Indicators)

Business Entities support defining KPIs — business metrics with target values, thresholds, and trend direction. KPIs appear in the Data Catalog and can be surfaced in SAC dashboards.

KPI propertyDescription
MeasureThe underlying measure the KPI is based on
TargetDesired value or formula for the target
ThresholdsGood/Warning/Critical bands for traffic-light coloring in SAC
TrendWhether higher is better (Revenue) or lower is better (Cost, Defects)

Calculated Attributes

Beyond simple column mappings, Business Entities support Calculated Attributes — new columns derived by formula at the semantic level. Similar to a calculated column in a view, but defined in the business layer without modifying the underlying technical model.

Business Layer

Consumption Models & Perspectives

Consumption Models assemble multiple Business Entities into a single, query-ready model. They are the most business-friendly consumption object in Datasphere.

Consumption Model Structure

A Consumption Model takes one Fact Business Entity as its center and associates multiple Dimension Business Entities to it — forming a semantic star schema. Additional properties like filters, variables, and currency settings can be defined at this level.

Perspectives

A Perspective is a subset of a Consumption Model tailored for a specific audience or use case. For example, a single "Sales" Consumption Model can have:

  • A Regional Manager Perspective — pre-filtered to the manager's region, showing only regional KPIs
  • A Finance Perspective — showing revenue and cost measures, hiding operational metrics
  • A Product Analysis Perspective — focused on product dimension drill-down

Perspectives reduce the need to build multiple separate models for different audiences.

Data Catalog

The Data Catalog automatically indexes all objects in Datasphere and provides a searchable, governed inventory of your data assets.

What gets cataloged automatically?

Every object created in Datasphere is automatically indexed by the catalog — no manual registration needed:

Local Tables Remote Tables Views Analytic Models Data Flows Replication Flows Business Entities Consumption Models Connections Spaces

Catalog Asset Page

Every cataloged asset has a detail page showing:

  • Overview: Technical name, business name, type, space, owner, dates
  • Schema/Structure: Columns with types, keys, measures
  • Lineage: Visual diagram of upstream sources and downstream consumers
  • Impact: List of objects that depend on this asset
  • Tags: Classification labels attached by data stewards
  • Glossary Terms: Business definitions linked to columns
  • Data Quality: Quality score and rule results
  • Sample Data: Preview of actual data (permission-dependent)

Business Glossary

The Glossary is a curated dictionary of business terms. Each term has a name, definition, owner, status (Draft/Approved/Deprecated), and can be linked to specific columns in catalog assets.

Example: Glossary term "Net Revenue" = "Total invoiced amount after all discounts and returns, excluding VAT, in transaction currency." This term is linked to the NETWR column in the SalesOrders table and the Revenue measure in the Sales Analytic Model.

Tags & Classifications

Tags are free-form labels for organizing and filtering catalog assets. Classifications are structured tags (hierarchical) for formal governance programs like GDPR or data sensitivity classification.

Data Quality Rules

Data quality rules run on local tables and produce a Data Quality Score (0–100%). Rules are defined on specific columns:

  • Completeness: % of non-null values in a column
  • Uniqueness: % of unique values (identifies duplicates)
  • Validity: % matching a defined pattern or range (e.g. date format, positive amounts)
Governance

Lineage & Impact Analysis

Lineage traces where data comes from. Impact shows what breaks when you change something. Together they are essential for governed data management.

Lineage Analysis (Upstream)

Lineage traces data backward through the dependency chain — from the consumer all the way to the original source system. The lineage diagram shows every table, view, flow, and model that data passed through.

Example lineage chain:
SAC Story → Analytic Model → Analytical Dataset View → Join View → Remote Table → S/4HANA ACDOCA table

This answers the question: "Where exactly does the Revenue figure in my SAC report come from?"

Impact Analysis (Downstream)

Impact analysis looks forward — given an object, what downstream objects depend on it? This answers: "If I change or delete this table, what reports/views/models will be affected?"

Critical before making changes: Always run Impact Analysis before modifying a source table schema (adding/removing columns, changing data types). A single column rename in a remote table can cascade and break dozens of downstream views and SAC stories.

Column-Level Lineage

Datasphere provides column-level lineage — not just object lineage. You can trace a specific column (e.g. "Gross Margin %") through every transformation and calculation to its source column(s). This is critical for audit and data certification.

Cross-Space Lineage

Lineage tracks objects across Spaces. If Space A's table is shared to Space B and used in a view there, the lineage diagram shows the cross-space dependency. This provides full tenant-wide visibility.

Lineage from External Sources

For objects originating from external connections (S/4HANA, BW, cloud storage), lineage shows the source system as the origin node — even though it's outside Datasphere. The connection metadata enables this cross-system tracing.

Governance

Security & Access Control

Datasphere provides multi-level security — from tenant-level user management down to row-level data access controls.

Security Layers

LayerControls whatManaged by
Tenant levelWho can log into the tenant, global admin rightsTenant Administrator
Space levelWho can access a Space and with which roleSpace Administrator
Object levelWhich objects are shared and visible across SpacesSpace Admin + sharing settings
Row levelWhich data rows a user can see (Data Access Control)DAC definition in Space

Data Access Controls (DAC) — Row-Level Security

A Data Access Control is a criteria-based filter automatically applied to queries based on the logged-in user's assignments. It restricts which rows of data a user can see, transparently — the user doesn't know the filter exists.

DAC Types

⊗
SQL-Based DAC

A view or table maps user IDs to permitted dimension values. E.g. user "JOHN" → allowed COMPANY_CODE = 1000 and 2000.

⊗
External Hierarchy DAC

Restrictions based on hierarchy node assignments. A manager is granted access to their org node, inheriting access to all subordinate nodes automatically.

DAC Assignment

DACs are attached to Analytical Datasets or Analytic Models. When SAC queries the model, Datasphere automatically appends the user's access filter before executing the query — no SAC-side configuration needed.

Interview tip: DAC in Datasphere is equivalent to Data Access Control in SAC — but enforced at the Datasphere engine level, before data reaches SAC. This is considered more secure as it cannot be bypassed by any SAC user, including story editors.

IDP Integration (SSO)

Datasphere integrates with any SAML 2.0-compatible Identity Provider (IdP) — Azure AD, Okta, SAP IAS (Identity Authentication Service). Users authenticate through their corporate SSO; Datasphere maps the IdP user to a Datasphere user account.

Audit Logging

Datasphere maintains audit logs for security-relevant actions — user logins, data access, configuration changes, connection usage. Audit logs can be forwarded to SAP Cloud ALM or SIEM tools for compliance monitoring.

SAC Integration

Live Connection to SAC

SAC connects to Datasphere via a Live Connection — real-time queries without data import into SAC. This is the standard consumption pattern.

Setting Up the Live Connection

1
In Datasphere — create a technical user

Create a dedicated technical user in Datasphere for SAC to connect with. Assign it DW Viewer role (minimum) in the relevant Space.

2
In SAC — create a new connection

Go to Connections → Add Connection → SAP Datasphere. Enter the Datasphere tenant URL, technical user credentials.

3
In Datasphere — expose objects

Ensure the Analytic Model or Analytical Dataset has "Expose for Consumption" turned ON.

4
In SAC — create a model using the connection

In SAC Modeler, create a new model → Live Data Model → select the Datasphere connection → browse and select the Analytic Model.

5
Build stories

The SAC model is now live — build stories, charts, and dashboards. Every chart query goes live to Datasphere's HANA engine.

What SAC Can Consume

Datasphere ObjectSAC ConsumptionNotes
Analytic ModelLive Data Model in SACPreferred — full star schema, hierarchies, variables
Analytical DatasetLive Data Model in SACSimpler, no full star schema assembly
Consumption Model (Business Layer)Live Data Model in SACMost business-friendly, includes KPIs
BEx Query (BW Bridge)Via BW Live connectionFor migrated BW queries during transition

Query Flow

When an SAC story user opens a chart:

  1. SAC translates the chart definition into an OLAP query
  2. Query is sent to Datasphere via the live connection (HTTPS)
  3. Datasphere's HANA engine executes the query — applying DAC filters, join logic, measure calculations
  4. Result set is returned to SAC and rendered in the chart
  5. No data is cached or stored in SAC — every interaction re-queries Datasphere
Performance implication: Live connections mean SAC performance is directly tied to Datasphere query performance. Slow Datasphere views = slow SAC stories. Use view persistency, partitioning, and proper aggregation to keep query times acceptable.
SAC Integration

Planning & Write-back

SAC Planning can use Datasphere Local Tables as the planning data store — enabling a modern, decoupled planning architecture.

Architecture Overview

In the traditional BPC architecture, planning data was stored inside BPC's own proprietary storage. The modern Datasphere+SAC architecture separates storage from UI:

  • Storage layer: Datasphere Local Tables (HANA Cloud)
  • Modeling layer: Datasphere views + Analytic Models
  • Planning UI: SAC Planning models and Input Tasks
  • Write-back: SAC writes plan data directly into Datasphere Local Tables

Setting Up Planning on Datasphere

1
Create a Local Table in Datasphere

Design the table structure to hold planning data — dimensions as columns (Company Code, Cost Center, Period, Version) and measure columns (Budget Amount, Quantity).

2
Build an Analytical Dataset on top

Create a view on the planning table. Set semantic type = Analytical Dataset. Define measures and dimension associations. Turn on Expose for Consumption.

3
In SAC — create a Planning Model

Create new Planning Model → Live Connection → SAP Datasphere → select the Analytical Dataset. SAC will map dimensions and measures from the Datasphere model.

4
Build Input Forms in SAC

Create SAC stories with Input Tasks / planning tables. Users enter plan data in SAC.

5
Write-back to Datasphere

When users submit/publish in SAC, plan data is written back to the Datasphere Local Table. The data is immediately available for reporting queries.

Datasphere + SAC vs BPC Architecture

AspectBPC (Standard / Embedded)Datasphere + SAC Planning
StorageBPC proprietary (InfoCube-like or AMDP)Datasphere HANA Cloud Local Tables
ModelingBPC dimensions, models, business rulesDatasphere views + Analytic Models
Planning logicScript Logic / Data ActionsSAC Data Actions + Transformation Flows
ReportingSAC or EPM Add-inSAC only
Integration with actualsVia BW extractorsDirect — actuals and plan in same HANA

Data Actions in this Architecture

SAC Data Actions still work in Datasphere-backed planning models. They execute against the Datasphere Local Table and can perform calculations, copy versions, allocate values across periods — same as BPC Script Logic but defined in SAC.

BW Bridge & Migration

BW Bridge enables running existing BW/4HANA objects natively within Datasphere — the transition path for organizations moving from BW to Datasphere.

What is BW Bridge?

BW Bridge is a special component of Datasphere that hosts a BW-compatible runtime. It allows organizations to lift and shift existing BW/4HANA objects into Datasphere without full re-development, while gradually migrating them to native Datasphere objects over time.

What BW Bridge Supports

BW ObjectBW Bridge supportMigration path
InfoObjects (Characteristics)Full support — run as-isConvert to Datasphere Dimension
InfoObjects (Key Figures)Full supportConvert to Datasphere Measure
Advanced DSO (aDSO)Full supportConvert to Local Table + Transformation Flow
CompositeProviderFull supportConvert to Datasphere View (Union/Join)
BEx QueriesFull support — consumable by SACConvert to Analytic Model
Transformations & DTPsSupportedConvert to Replication/Transformation Flows
Process ChainsPartial — basic chainsRe-build as Task Chains
ABAP routines in transformationsLimited — simple routines onlyRe-implement in SAP HANA SQL/scripting

BW Bridge Migration Approach

1
Assess BW landscape

Inventory all BW objects. Identify which are actively used. Use SAP's Migration Assessment tool to get a complexity rating per object.

2
Provision BW Bridge Space

Create a dedicated BW Bridge Space in Datasphere. This space has BW runtime capabilities.

3
Transfer BW objects

Use the BW-to-Datasphere transfer tools to move BW objects into the BW Bridge Space. This is largely automated for standard objects.

4
Validate and connect SAC

Connect SAC to BW Bridge via BW Live Connection. Validate that existing BEx Query-based SAC stories still work.

5
Gradual native migration

Over time, re-build BW objects as native Datasphere objects (Replication Flows replace DTPs, SQL Views replace transformations, Analytic Models replace BEx Queries).

BW Bridge is a bridge, not a destination. SAP's recommendation is to eventually migrate all objects to native Datasphere. BW Bridge provides continuity during the transition, but long-term all new development should be in native Datasphere.
Advanced Topics

Currency Conversion

Datasphere has built-in currency conversion using SAP's standard TCURR/TCURF exchange rate tables, enabling multi-currency reporting without manual conversion logic.

How It Works

When a measure is tagged with a currency column (e.g. transaction currency WAERS), Datasphere can automatically convert amounts to a target currency at query time using exchange rates stored in the system.

Exchange Rate Tables

Datasphere uses the standard SAP exchange rate tables, typically replicated from S/4HANA:

TableContent
TCURRExchange rates — from/to currency pairs with date validity
TCURFExchange rate factors (for currencies with large rate differences)
TCURXCurrency decimal places
TCURNExchange rate type descriptions (M=average, B=buying, S=selling)

Currency Conversion Configuration

On a measure in an Analytic Model, you configure:

  • Source currency column: The column containing the transaction currency code (e.g. WAERS)
  • Target currency: Fixed (e.g. always EUR) or variable (SAC user chooses at runtime)
  • Exchange rate type: M (monthly average), B (bank buying), S (bank selling), or custom
  • Date for rate lookup: Fixed date, or dynamic (e.g. posting period date column)
Interview tip: Currency conversion in Datasphere is equivalent to translation settings in a BPC model or the CURRENCY_TRANSLATION function in BW/HANA. The mechanics are the same — exchange rate type, source/target currency, and a reference date determine which TCURR entry is used.

Performance & Partitioning

Query performance in Datasphere depends on data modeling, storage choices, and HANA-specific optimization techniques.

Performance Levers

TechniqueWhen to useEffect
View PersistencyComplex multi-join views used in SAC reportsMaterializes view — query reads snapshot instead of re-computing
Table PartitioningLarge local tables queried with date/period filtersPartition pruning — only relevant partitions scanned
Replica TableRemote tables queried frequently in reportsQuery hits local HANA instead of remote source
Table IndexingColumns used in JOIN conditions or WHERE clausesFaster lookup — avoids full table scan
Aggregation nodeFact views with high row countsPre-aggregates before presenting to SAC
Column StoreDefault in HANA — analytical queriesColumnar compression + parallel scan

Monitoring Tools

  • Space Monitor: CPU, memory, disk usage per Space. Identify resource-intensive flows or queries.
  • Data Integration Monitor: Status and logs for all Replication Flows, Data Flows, and Task Chains. Shows run duration, row counts, errors.
  • Database Explorer (HANA): Open via Datasphere to run SQL queries, inspect execution plans, view column statistics.
  • SAC Performance Monitor: In SAC, story-level query performance metrics showing which queries are slow.

Common Performance Anti-Patterns

Anti-pattern 1 — Stacking virtual views: Building 5+ levels of virtual views (view on view on view) creates deep execution plans. Use view persistency at intermediate levels or flatten the logic into a single view.
Anti-pattern 2 — Federation for reporting: Using remote (federated) tables as the base for SAC stories hits the source system for every user interaction. Replicate first, then report.
Anti-pattern 3 — Missing join cardinality: Wrong cardinality settings in graphical view joins mislead the HANA optimizer. Always set correct cardinality (1:N, N:1 etc.).

Open SQL Schema & APIs

Datasphere exposes multiple programmatic interfaces for external tools and custom integration — Open SQL, OData, and REST APIs.

Open SQL Schema

Every Space can optionally expose an Open SQL Schema — a direct HANA SQL access point to the space's objects. External tools connect via standard JDBC/ODBC.

  • Query views and tables directly using HANA SQL
  • Compatible with SAP HANA Studio, DBeaver, Tableau, Power BI, Excel (via ODBC)
  • Read-only — cannot INSERT/UPDATE/DELETE via Open SQL
  • Access is subject to Datasphere role/DAC restrictions
-- Connect via JDBC, then query a Datasphere view
SELECT "PRODUCTID", "SHORT_DESCR", "PRICE"
FROM "MY_SPACE"."ProductCatalogView"
WHERE "PRODCATEGORYID" = 'LT';

OData API

Datasphere exposes OData v4 endpoints for consuming data programmatically. When an Analytical Dataset or Consumption Model has "Expose for Consumption" enabled, it can be accessed via OData.

GET https://<tenant>/api/v1/datasphere/spaces/MY_SPACE
    /objects/SalesAnalyticModel
    ?$select=PERIOD,AMOUNT&$filter=YEAR eq '2024'&$format=json

Datasphere provides a Management REST API for administrative operations — automating space creation, user management, flow execution, and catalog operations. Useful for CI/CD pipelines and automated deployment.
